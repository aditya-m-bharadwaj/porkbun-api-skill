# Sister-project scaffold for `porkbun-api-skill` (no CLI yet)

This session bootstrapped the directory layout, AI-facing prompts, memory layer, and graphify integration for the Porkbun-API skill. The CLI itself (`bin/porkbun-api-skill`) is **not** built yet — only a placeholder stub. The next session is the build.

Sister project for reference at every turn: <https://github.com/aditya-m-bharadwaj/linode-api-skill>. Mirror its architecture; adapt for Porkbun's dual-key auth, idempotency-key support, and registrar-domain semantics.

## What was done

### Scaffolding files

- `CLAUDE.md` — project-level hard rules + memory protocol + commit-message rules. Identical structure to linode-api-skill's CLAUDE.md, adapted for Porkbun:
  - Both `apikey` *and* `secretapikey` must never enter AI context.
  - Domain-registration calls require a fresh price quote echoed to the user, and a `--cost-cents <N>` flag matching the quote.
  - Every write call must carry an `Idempotency-Key` (auto-attached by the CLI).
- `.claude/skills/porkbun-api-skill/SKILL.md` — full skill contract. Five-tier classifier (read / mutating / destructive / billable / privilege — no separate `financial` because Porkbun's API doesn't have payment-method endpoints). Per-resource confirmation playbooks for domains, DNS, SSL, account, API keys, email. SSL retrieve is `privilege` (returns private key material) — write to disk mode 0600, do not echo to chat.
- `.claude/commands/resume.md`, `.claude/commands/save.md` — project-level slash commands matching the memory protocol. The `/save` command codifies "commit docs alongside code, not after" (carried over from operator preference set in linode-api-skill on 2026-05-10).
- `docs/README.md` — human-facing memory-layer index, mirrors linode-api-skill's docs/README.md.
- `docs/decisions/TEMPLATE.md`, `docs/progress/TEMPLATE.md` — copied / adapted from sister project.

### Seed ADRs

Eight ADRs encoding decisions already made (carried over) or made fresh for Porkbun:

- `0001-stdlib-only-python` (carry-over): stdlib-only Python 3.8+, no third-party deps.
- `0002-safety-classifier-five-tiers` (porkbun-specific): five tiers instead of linode-api-skill's six — no separate `financial` tier because Porkbun has no payment-method endpoints. POST-to-delete routes are explicit destructive entries (no method-based default catches them).
- `0003-cross-platform-credential-pair-storage` (porkbun-specific): dual-key (`apikey` + `secretapikey`) stored as a single JSON blob in macOS Keychain / libsecret / file fallback. Atomic — never rotate one without the other.
- `0004-ai-safe-credential-entry-gui-dialog` (porkbun-specific): one native OS dialog captures *both* secrets, validates against `/ping`, stores atomically. Adapted from linode-api-skill's single-token pattern.
- `0005-prompted-by-trailer-convention` (carry-over): `Prompted-By:` + `Co-Authored-By:` trailers on AI-shaped commits.
- `0006-monolithic-cli-file` (carry-over): single-file CLI at `bin/porkbun-api-skill`.
- `0007-versioning-semver` (carry-over): SemVer 2.0.0 with `-alpha.N` / `-beta.N` / `-rc.N`.
- `0008-idempotency-key-on-write-ops` (porkbun-specific, defining decision): `_request` auto-attaches `Idempotency-Key: <uuid4>` on every POST. Operators may override via `--idempotency-key <key>` or opt out via `--no-idempotency-key` (debug only).

## What's next (recommended pickup)

The next session should **build the CLI**, following the SKILL.md contract and the ADRs as the design spec. Order:

1. **HTTP + credential layer.** `_request` with auto-`Idempotency-Key` insertion (ADR-0008). Credential storage as JSON blob in Keychain / libsecret / file (ADR-0003). `_validate_path` + path normalization (handle both `/v3/...` and `/...`).
2. **Classifier.** `_normalize_path`, `classify(method, path)`, the five-tier matrix. Populate `_DESTRUCTIVE_EXACT`, `_BILLABLE_EXACT`, `_PRIVILEGE_PREFIXES` from `docs/api-spec.json`. Default fallback: GET → read, POST → mutating. **No** method-based default for destructive (POST-to-delete is explicit-only).
3. **Named commands.** `whoami`, `balance`, `domains`, `domain <name>`, `price <name>`, `register`, `renew`, `transfer`, `auto-renew`, `nameservers`, `dns {list,add,edit,delete}`, `ssl`, `audit-log`, `classify`. The billable trio (`register`, `renew`, `transfer`) re-fetches `/domain/checkDomain` and refuses if quoted price ≠ `--cost-cents`.
4. **Generic `api` gateway.** Same flag matrix as linode-api-skill.
5. **`gui-setup`.** Dual-field dialog (osascript → zenity → kdialog → Get-Credential). Validate against `/ping`, store atomically.
6. **Tests.** Mirror linode-api-skill's test_classify.py — offline-only, load script via `SourceFileLoader`, exercise classifier + path validation + helper functions.
7. **Audit log.** `~/.porkbun-api-skill/audit.log` mode 0600. One JSON line per mutation: timestamp, user, action, target, idempotency key, classification, body keys (never values, never credentials).

When that lands, a `/save` should be done **alongside** that commit (or commit batch) — not as a trailing docs commit.

## Open questions

- **GitHub repo creation.** When (today/now? after first build session?) and under whose namespace? Install scripts currently point at `aditya-m-bharadwaj/porkbun-api-skill` as a placeholder.
- **SSL bundle output path.** SKILL.md says write to `~/.porkbun-api-skill/ssl/<domain>.<timestamp>.{certificatechain,privatekey,publickey}.pem`. The `<timestamp>` part means a fresh retrieve doesn't overwrite the old one; the user has to clean up periodically. Alternative: overwrite each time and only keep the latest. Operator preference TBD.
- **`/account/invite` classification.** Currently `privilege` (it grants account access). But Porkbun's invite system may issue a sub-user with a separate credential pair, in which case the new invitee's actions are isolated. Skim the spec for `inviteStatus` response shape during the build session.
- **Idempotency-key behavior on read-style POSTs.** The spec says these endpoints don't cache; the header is harmless. But the CLI could skip attaching it on read POSTs for cleanliness. Trade-off: simpler `_request` (always attach) vs. cleaner request log. Current design: always attach.
- **Should the CLI offer a `linode-api-skill`-style `--cold` for `transfer`?** Domain transfers have a multi-day in-flight period during which the originating registrar can reject. Not really a "cold/warm" issue; just a UX consideration for status display.
